## 1) 产品身份（不可动摇）

Aria 是一个可下载、可部署、可运营的独立产品，而不是任何开发代理的“外壳”。

她必须具备：

1. 独立人格：稳定人格核心、关系记忆、边界一致性  
2. 独立运营：无需依赖 Codex / Antigravity / Aria Kernel 在线介入  
3. 独立修复：运行异常可自动检测、自修复、可回滚  
4. 独立升级：策略和能力可在线升级，不中断服务  
5. 独立成长：基于用户互动持续学习，不需要人工逐条教

---

## 2) 交付形态（明确）

### 2.1 客户端分发（必须）

1. macOS：`.dmg` / 签名 `.app`
2. Windows：`.msi` / `.exe`
3. iOS：TestFlight / App Store 包线

### 2.2 灵活部署（必须）

1. 本地单机部署（开发/私有演示）
2. Docker 部署（标准化交付）
3. 云端部署（可运营环境，支持灰度）

---

## 3) 三方能力“注入”规则（不是“依赖”）

### 3.1 注入原则

1. Codex、Antigravity、Aria Kernel 的能力要以“产品能力模块”形式落入 Aria
2. 模块化后必须可在 Aria Runtime 内独立运行
3. 禁止出现“离开某个开发代理即不可运行”的单点依赖

### 3.2 能力映射

1. Codex -> 规划与执行内核（Planner + Task Orchestrator）
2. Antigravity -> 情绪体验内核（Emotion UX + Aesthetic Tokens）
3. Aria Kernel -> 自主调度内核（Autonomy Runtime + Policy Engine）

---

## 4) 架构约束（强制）

1. 人格层（Soul Layer）与执行层（Execution Layer）分离
2. 自治内核（Autonomy Kernel）必须具备：
   - Tick 调度
   - 自主建议队列
   - 自修复钩子
   - 在线策略热更新
3. 客户端必须只依赖稳定 API，不依赖开发时工具链
4. 所有关键能力必须有状态可观测接口

---

## 5) DoD（此方向下的完成标准）

一个“独立能力”只有同时满足以下条件才算完成：

1. 可运行：在 Aria Runtime 内可独立执行  
2. 可恢复：异常可自修复或回滚  
3. 可升级：参数/策略可热更新  
4. 可观测：有状态接口与日志  
5. 可验证：有 smoke 命令与验收记录

---

## 6) 禁止项

1. 禁止把“开发者能力”直接写死为运行时依赖
2. 禁止出现“需要人工值守才能工作”的关键链路
3. 禁止“只讲概念不落代码”的伪分工
4. 禁止把人格能力做成漂移严重的短期提示词拼接

---

## 7) 当前执行口令（对三方）

1. 所有任务先写执行包，再写代码，再给验收命令
2. 没有代码改动不算开始；没有验证结果不算完成
3. Codex 对三方结果统一验收，不通过不并线
