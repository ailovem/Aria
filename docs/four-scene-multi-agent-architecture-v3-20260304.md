# Aria 四场景多 Agent 后端架构（V3｜对齐 Aria Kernel 机制）

更新时间：2026-03-04  
目标：把“工作 / 娱乐 / 生活 / 情感”四场景做成真正可并行、可自治、可演进、可恢复的独立产品内核。

---

## 0) 一句话结论

不是做 4 套后端，而是做 **1 个控制平面 + 4 个场景分身 Agent + 1 套共享大脑与记忆底座**。

> 用户一句话 → 控制平面判定场景与目标 → 主 Agent 编排任务图 → 场景 Agent 并行执行 → 统一交付与记忆沉淀。

---

## 1) 为什么 Aria Kernel 自由度高（可复用到 Aria）

基于官网和本机系统文件，Aria Kernel 的高自由度来自 6 件事：

1. **单网关多 Agent 路由绑定**
   - 一个 Gateway 承载多个 Agent，按 channel/account/peer 路由。
   - 路由是配置驱动，不是写死在业务代码里。

2. **Agent 隔离 + 协作总线**
   - 每个 Agent 拥有独立 workspace、session、短期记忆。
   - 协作通过工单总线（handoff-bus），而非互相串上下文。

3. **队列与重试是第一等公民**
   - 命令队列支持背压、退避、重试预算和降级。
   - 失败可追踪、可重放，不是“失败就丢”。

4. **模型路由与回退机制工程化**
   - 主模型 + fallback 链路 + 超时预算 + 失败分类。
   - 高并发下保持服务连续性。

5. **自愈与自升级流程自动化**
   - 定时体检 + 锁机制 + 冷却时间 + 失败恢复。
   - 升级流程有探测、回滚、报告，不靠人工盯。

6. **治理中台与生产真源分离**
   - 生产库（运行真源）与治理库（规则、索引、报告）职责分离。
   - 避免“管理文档覆盖运行数据”。

## 2) Aria 目标架构（四场景四分身）

## 2.1 控制平面（Control Plane）

职责：
- 接收所有入口（PC/iOS/飞书/语音）的用户意图。
- 识别场景与风险等级。
- 编译任务图（Task Graph）。
- 调度 4 个场景 Agent，并统一回执。

核心组件：
- **Intent Compiler**：自然语言 → 结构化任务。
- **Scene Router**：路由到 work/fun/life/love。
- **Planner**：单步/多步 DAG 生成。
- **Execution Orchestrator**：并行执行 + 依赖控制。
- **Timeline Ledger**：提交-执行-重试-完成全链路记录。

## 2.2 四个场景 Agent（分身）

### A. WorkAgent（工作分身）
- 目标：产出、交付、自动化办公。
- 能力：文件检索、代码执行、文档生成、跨系统同步。
- 重点指标：任务完成率、时效、可验证交付。

### B. FunAgent（娱乐分身）
- 目标：互动乐趣、小游戏、轻创作。
- 能力：小游戏生成/运行、社交互动、轻量内容创作。
- 重点指标：互动时长、复玩率、低打扰。

### C. LifeAgent（生活分身）
- 目标：生活管理、家庭与设备协同。
- 能力：日程提醒、相册整理、消息摘要、家庭任务协作。
- 重点指标：提醒命中率、生活任务闭环率。

### D. LoveAgent（情感分身）
- 目标：实时陪伴、情绪共情、关系维护。
- 能力：情绪识别、记忆触达、关系节律管理。
- 重点指标：回应质量、关系记忆准确率、陪伴连续性。

## 3) 共享“大脑 + 记忆 + 技能”底座

## 3.1 共享大脑（Brain Kernel）

不是每个场景各自“再造一套模型”，而是共享以下内核：

1. **Model Router**：任务级路由（推理型 / 对话型 / 工具型）。
2. **Policy Engine**：风险、权限、预算、节流策略。
3. **Tool Runtime**：统一工具协议（入参/出参/幂等键/审计证据）。
4. **Learning Loop**：基于反馈更新策略权重，不改用户记忆真源。

## 3.2 记忆系统（Memory OS）

采用“共享主干 + 场景分区 + 人格核心”的三层结构：

- **Layer-0 人格核心记忆（Soul Core）**
  - 用户称呼、关系边界、长期互动基调。
  - 所有场景可读，只有治理模块可写。

- **Layer-1 场景记忆命名空间（Scene Memory Namespace）**
  - `memory_work/*`、`memory_fun/*`、`memory_life/*`、`memory_love/*`
  - 各场景默认仅写自己的短/中期记忆。

- **Layer-2 跨场景知识层（Knowledge Graph + Vector）**
  - 可复用事实、技能、流程模板。
  - 通过“事实提炼”后进入共享知识，不直接写原始聊天。

## 3.3 数据库与向量库建议

### 事务库（PostgreSQL）
- 用途：任务、会话、消息、工单、权限、审计。
- 关键表：
  - `users`
  - `sessions`
  - `messages`
  - `tasks`
  - `task_steps`
  - `task_events`
  - `agent_handoffs`
  - `policy_snapshots`
  - `tool_runs`

### 向量库（Qdrant / pgvector 二选一，推荐 Qdrant）
- Collection 切分：
  - `soul_core`
  - `scene_work_memory`
  - `scene_fun_memory`
  - `scene_life_memory`
  - `scene_love_memory`
  - `skills_knowledge`
- 检索策略：Hybrid（向量 + 关键词 + 时间衰减 + MMR）。

### 对象存储（S3 / MinIO）
- 用途：图片、语音、PDF、任务产物、报告附件。
- 只保存对象索引到 PostgreSQL，不把大文件写入事务库。

### 缓存与队列（Redis + JetStream/Kafka）
- Redis：会话热缓存、幂等键、短锁。
- 消息总线：任务事件流（创建、执行、重试、补偿、完成）。

## 4) 多 Agent 并行调用设计（核心）

## 4.1 协同协议（必须落地）

每次跨 Agent 协作必须有工单契约：

- `ticket_id`
- `goal`
- `inputs`
- `constraints`
- `acceptance_criteria`
- `deadline`
- `rollback_rule`

禁止“把整个聊天上下文直接丢给另一个 Agent”。

## 4.2 执行模型

1. 主 Agent（Scene Owner）拿到用户目标。  
2. Planner 生成 DAG（可并行步骤 + 依赖步骤）。  
3. Orchestrator 按优先级投递队列。  
4. Worker/Tool Runner 执行并回传证据。  
5. Reviewer Gate 校验（质量/风险/完整性）。  
6. Delivery Composer 生成对用户可读回执。  

## 4.3 幂等与补偿

- 每个步骤必须带 `idempotency_key`。
- 重试不应造成重复发送/重复扣费/重复写入。
- 对外部副作用操作（发消息、写三方系统）必须有补偿动作（Saga）。

---

## 5) 稳健性设计（防卡、防乱、防漂移）

## 5.1 Runtime Guardian（Aria 必做）

- 互斥锁：防止多份守护进程并发互踢。
- 健康探针：Gateway/API/Bridge/Queue/DB/Vector 分层探测。
- 冷却窗口：防止反复重启造成雪崩。
- 自愈报告：每次修复落盘，可审计。

## 5.2 失败分类（统一四层）

- `network`：外网/内网连通异常
- `permission`：权限不足或用户未授权
- `policy`：策略拦截（白名单、风控、预算）
- `execution`：工具报错、脚本失败、超时

## 5.3 SLO（首版建议）

- 聊天首 token：P95 < 2.5s
- 自动任务启动：P95 < 4s
- 任务成功率：> 92%
- 失败可解释率：> 99%
- 关键任务可重放率：100%

---

## 6) 安全与边界（防止“高自由度失控”）

1. **能力分级授权**：读、写、执行、外发四级。
2. **扩展三件套**：白名单 + 签名校验 + 能力沙箱。
3. **密钥托管**：密钥仅在 secrets/vault，禁止写入业务日志。
4. **高风险二次确认**：支付、删除、外发敏感信息。
5. **审计链**：所有工具调用必须可追溯到 `task_id` 与操作者。

## 7) 面向 Aria 当前代码的落地路线（不是空文档）

## P0（1-2 天）— 先稳态

- 建立 `scene_agent_topology` 配置（4 场景与能力边界）。
- 在统一时间线里补 `agent_id`、`ticket_id`、`scene` 字段。
- 接入 Runtime Guardian v1（锁、探针、冷却、自愈报告）。

验收：
- 任何任务都能看到“由哪个分身执行、执行到哪一步、失败原因是什么”。

## P1（3-5 天）— 真并行

- 上线 DAG Planner + Queue Lane（scene lane + priority lane）。
- 支持并行步骤与依赖步骤混合执行。
- 场景间协作通过工单，不共享原始上下文。

验收：
- “找资料→生成摘要→发飞书→同步手机”可并行与串行混合执行。

## P2（5-7 天）— 记忆与学习

- 记忆分层写入策略（短期->复核->长期）。
- 向量检索加时间衰减和事实置信度。
- 场景偏好权重可在线调优并版本化回滚。

验收：
- 不同场景回复风格稳定，且能调用跨场景事实但不串私密上下文。

## P3（持续）— 自升级与生态

- 插件市场接入签名机制、沙箱配额、版本灰度。
- 自升级支持“健康探针通过后切流”。
- 每次升级自动生成变更与回滚记录。

---

## 8) 你关心的“独立运营”如何保证

要做到“脱离开发者也能跑”，必须满足：

1. **配置驱动**：模型、策略、场景、路由都可配置，不写死在代码。  
2. **执行留痕**：所有动作有证据，便于自动复盘和自修复。  
3. **版本可回滚**：策略/模型/插件都能一键回退。  
4. **守护自恢复**：服务崩了自动拉起，异常可被主动发现。  
5. **最小人工介入**：运维动作尽量脚本化和策略化。
