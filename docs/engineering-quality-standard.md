# Aria 工程质量基线（Engineering Quality Standard）

更新时间：2026-03-03  
Owner：Codex（项目负责人）

## 1) 目标

保证每次交付都达到“可用、可维护、可扩展”，禁止敷衍开发。

## 2) 文件级最低标准（必须全部满足）

1. **可运行**：新增/修改代码可构建、可启动、可调用
2. **可理解**：结构清晰、命名清晰、无无意义复杂度
3. **可验证**：有明确验证命令或手工验证路径
4. **可追踪**：能说明改动原因、影响范围、回滚方式

## 3) 提交前检查（DoD Checklist）

1. 功能验收通过（核心流程可用）
2. 构建通过（至少一条构建链路）
3. 文档同步（README/协议/说明）
4. 风险说明（已知问题 + 后续计划）

## 4) 禁止项

1. 空壳函数和未实现逻辑伪装为完成
2. 只改 UI 不改行为却宣称功能完成
3. 未验证直接交付
4. 用临时硬编码替代明确配置且无说明

## 5) 汇报模板（固定）

每次交付必须包含：

1. 改动文件
2. 核心设计
3. 验证结果
4. 剩余风险
