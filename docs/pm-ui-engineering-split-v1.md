# Aria 前后端与 UI 分工防重执行单（v1）

更新时间：2026-03-03
负责人：Codex

## 1. 原则（防重复）
1. 任何需求先落在「API 契约 + 状态机」再落 UI。
2. UI 仅消费 ViewModel，不直接拼业务逻辑。
3. 一个模块只允许一个 Owner 改逻辑：
   - 逻辑 Owner 可改 `apps/desktop/src/App.tsx` 的状态与动作。
   - UI Owner 只改 `apps/desktop/src/styles.css` 与纯展示组件。
4. 禁止双人同时改同一职责文件；若必须，先拆成两个文件再并行。

## 2. 本阶段职责划分

### Codex（产品/架构/工程负责人）
- 负责信息架构与交互主流程（3 大块 IA、场景导航、模块跳转逻辑）。
- 负责前端状态机与后端契约定义。
- 负责“开箱即用”路径与错误可理解提示。
- 负责验收标准与里程碑质量门禁。

### UI 同事（视觉与交互实现）
- 负责视觉规范、卡片排版、动画节奏、色彩与可读性。
- 负责移动端自适应与组件一致性。
- 负责无障碍层（可点击区、焦点态、字号层级）。
- 不改后端接口、不改业务状态机。

## 3. 文件级所有权（硬约束）
- `apps/desktop/src/App.tsx`：Codex（业务流与状态机）
- `apps/desktop/src/api.ts`：Codex（API 契约与错误处理）
- `apps/desktop/src/styles.css`：UI 同事（视觉实现，必要时由 Codex 提交结构配套）
- `docs/*.md`：Codex（产品与架构文档）

## 4. 交付节奏（避免重复开发）
1. Codex 先提交「结构+数据契约」版本。
2. UI 同事只在结构稳定后提交视觉增强版本。
3. 合并前由 Codex 对照检查：
   - 是否越权改了对方职责文件。
   - 是否引入重复按钮/重复模块入口。
   - 是否与 API 契约不一致。

## 5. 当前已执行到位
1. 前端已重构为三大块：
   - 左侧：脑系统、自动化与技能中心（含进入配置/上传/编辑）
   - 中部：女友形象互动
   - 右侧：按场景导航（工作/娱乐/生活/情感）
2. 已统一“场景 -> 模块面板”映射，防止不同入口跳到重复页面。
3. 已保留 6 模块超级脑图，作为统一脑系统入口。
4. 连接失败提示已改为可执行文案（先启动 runtime 再同步）。

2. 把“上传资料/编辑规则”接到真实 API（Memory/Policy/Automation）。
